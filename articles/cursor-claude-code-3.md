## 1. CursorとClaude Codeはそもそも何が違うのか？

結論から言うと、設計思想がまったく異なります。

**Cursorとは**、VS Codeをフォークして作られたAI統合エディタです。エディタの中にAIが住んでいる感覚で、Tab補完、インライン編集、マルチファイル参照がシームレスに動きます。

**Claude Codeとは**、Anthropic社が提供するCLIベースのAIエージェントです。ターミナルで動作し、ファイルの読み書き、コマンド実行、Git操作まで自律的に行います。

| 比較軸 | Cursor | Claude Code |
|-------|--------|-------------|
| 動作環境 | GUIエディタ（VS Code系） | ターミナル（CLI） |
| 操作モデル | エディタ内でインライン編集 | 自然言語で指示→自律実行 |
| コンテキスト | 開いているファイル + @参照 | プロジェクト全体を自動探索 |
| 得意領域 | 1-3ファイルの素早い編集 | プロジェクト横断の大規模変更 |
| Git連携 | 差分プレビュー | コミット・PR作成まで自動 |
| 学習曲線 | 低い（VS Codeユーザーなら即日） | 中程度（CLAUDE.mdの設定で真価を発揮） |

一言で表すなら、Cursorは「超優秀な副操縦士」、Claude Codeは「自律型のシニアエンジニア」です。

## 2. インライン編集の体験はどちらが上か？

正直に言います。インライン編集はCursorが圧倒的に快適です。

Cursorの`Cmd+K`で関数を選択して「この関数をメモ化して」と入力すると、2秒で差分が表示されます。Accept/Rejectをワンクリック。この体感速度はClaude Codeでは実現できません。

```javascript
// Cursorでの典型的な使い方：関数を選択して Cmd+K
// 「この関数にエラーハンドリングを追加して」と入力するだけ

// Before
async function fetchUser(id) {
  const res = await fetch(`/api/users/${id}`);
  return res.json();
}

// After（Cursorが2秒で生成）
async function fetchUser(id) {
  try {
    const res = await fetch(`/api/users/${id}`);
    if (!res.ok) throw new Error(`HTTP ${res.status}`);
    return res.json();
  } catch (err) {
    console.error(`Failed to fetch user ${id}:`, err);
    throw err;
  }
}
```

Tab補完も同様です。変数名を途中まで打つと、周囲のコードを理解した上で残りを補完してくれます。型定義との整合性も高い。

ただし、Cursorのインライン編集には限界があります。3ファイル以上にまたがる変更になると、1ファイルずつ手動で適用する必要があり、効率が急激に落ちます。

## 3. 大規模リファクタリングではどちらが強いのか？

ここがClaude Codeの独壇場です。

先月、APIルーターのリファクタリング（665行→271行への分割）を行いました。ルーターからサービス層を抽出し、テストを移行し、インポートを全ファイルで更新する作業です。

```bash
# Claude Codeでの実際の指示
claude "services/api/app/routers/workflows.py が665行あって肥大化している。
ビジネスロジックを services/api/app/services/workflow_service.py に抽出して、
ルーターは薄いラッパーにして。テストも合わせて修正して。"
```

Claude Codeはこの1つの指示で以下を実行しました：

1. `workflows.py`を読み、ビジネスロジックを特定
2. `workflow_service.py`を新規作成
3. ルーターをリファクタリング（665行→271行）
4. テストファイルのインポートを修正
5. 他ファイルからの参照を検索・更新

所要時間：約4分。手動なら2-3時間はかかる作業です。

同じことをCursorでやろうとすると、ファイルごとにComposer（マルチファイル編集機能）で指示を出す必要があり、ファイル間の整合性は自分で確認しなければなりません。

## 4. 実測データで比較するとどうなるか？

3ヶ月間の業務で計測した数値です。完全に主観排除はできませんが、傾向は明確に出ています。

| 指標 | Cursor | Claude Code |
|-----|--------|-------------|
| 小規模修正（1-2ファイル）の速度 | 約40行/分 | 約25行/分 |
| 大規模変更（5+ファイル）の速度 | 約15行/分 | 約60行/分 |
| Tab補完の採用率 | 約70% | N/A（補完機能なし） |
| 生成コードの初回正答率 | 約75% | 約85% |
| アーキテクチャ提案の品質 | △（局所最適になりがち） | ◎（全体構造を踏まえる） |
| テスト自動生成の精度 | ○ | ◎（既存テストのパターンを踏襲） |

注目すべきは「大規模変更の速度」の逆転です。Cursorは小さな編集では速いですが、ファイル数が増えるとClaude Codeの方が4倍速くなります。

これはアーキテクチャの違いが直接影響しています。CursorはエディタのUIを通じた操作なので、ファイルが増えるほどオーバーヘッドが増えます。Claude Codeはターミナルで一括処理するため、ファイル数に対してスケールします。

## 5. 料金はどちらがコスパ良いのか？

2026年3月時点の料金比較です。

| プラン | Cursor | Claude Code |
|-------|--------|-------------|
| 月額 | $20（Pro） | $20（Claude Pro）/ $100（Max） |
| モデル | Claude 3.5 Sonnet等を選択可 | Claude Sonnet / Opus |
| 使用制限 | 500回/月（Pro） | 使用量ベース |
| 追加費用 | $40（Ultra、無制限） | API従量課金も選択可 |

私の使い方（1日2-3時間の開発）では、Cursor Pro（$20）+ Claude Code Max（$100）で月$120です。高いと感じるかもしれませんが、時給換算すると毎月20-30時間は節約できているので、十分にペイしています。

コストを抑えたいなら、まずCursor Proだけで始めて、大規模タスクが増えた段階でClaude Codeを追加するのが現実的です。

## 6. 実際の使い分けワークフローとは？

3ヶ月の試行錯誤を経て、現在はこう使い分けています。

**Cursorを使う場面：**
- 既存コードの小修正（バグ修正、型の追加、変数名変更）
- 新しいコンポーネントの雛形作成
- コードレビュー中にインラインで修正を入れる
- Tab補完による日常的なコーディング速度の底上げ

**Claude Codeを使う場面：**
- 5ファイル以上にまたがるリファクタリング
- 新機能のアーキテクチャ設計と実装
- テストの一括生成
- CI/CDスクリプトやインフラ設定の変更
- `CLAUDE.md`にプロジェクトルールを書いて品質を統一

```bash
# 私の1日のワークフロー
# 朝：Claude Codeで大きなタスクを片付ける
claude "昨日のPRレビューコメントをすべて対応して"

# 日中：Cursorで細かい実装を進める
# Cmd+K、Tab補完、インライン修正

# 夕方：Claude Codeでテストとドキュメントを整備
claude "今日変更したファイルのテストを書いて"
```

重要なのは、両方を「同じプロジェクトで」使うことです。CursorのエディタでコードをブラウズしながらClaude Codeに大きなタスクを投げる。この組み合わせが最も生産性が高いと実感しています。

## まとめ

3ヶ月使い比べた結論はこうです。

- **小さな編集・日常コーディング** → Cursorが快適
- **大規模変更・アーキテクチャ作業** → Claude Codeが圧倒的
- **最強の組み合わせ** → 両方を用途別に使い分ける

「どちらか片方だけ」と聞かれたら、Claude Codeを選びます。理由は、プロジェクト全体を理解した上でのコード生成品質がCursorより明確に高いからです。ただし、Cursorの編集体験は捨てがたい。だから両方使っています。

まだどちらも試していない方は、まずClaude Codeの無料枠で触ってみてください。ターミナルから`claude`と打つだけで、AIエージェントによる開発がどういうものか体感できます。

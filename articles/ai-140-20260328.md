## 1. Anthropicが公式に認めた「AI SREの限界」

2026年3月19日、QCon Londonの講演でAnthropicは興味深い実験結果を発表しました。

**実験内容**: ClaudeにClaudeのインフラログとメトリクスを読ませ、障害の原因特定と修復を自動実行させる。

**結果**: 失敗。

具体的には、「メモリ使用量が急増した直後にレスポンスタイムが悪化した」というログを見て、Claudeは**メモリが原因**と判断しました。しかし実際の根本原因は**クエリパターンによるロック競合**でした。

これが「相関→因果誤認」問題です。

人間のSREなら、過去に似たパターンを経験していたり、「先週デプロイしたクエリ最適化がロックの取り方を変えた」という文脈を持っています。LLMにはこの**暗黙知**がありません。

Anthropicはこの結果を踏まえ、AIのSRE活用は「判断を任せる」のではなく「整理を支援する」用途が現実的だと結論づけています。

### なぜこれが重要か

SREはAI活用の中でも難易度が高い分野ですが、この問題はSREに限りません。**あらゆる業務自動化で同じ構造の失敗が起きます。** コードレビュー、ドキュメント生成、テスト設計——AIが「もっともらしい回答」をするほど、間違いに気づきにくくなる。

これを知らずに「全部AIに任せよう」とすると、静かに品質が崩壊します。

## 2. 140エージェント運用で見えた「AIが得意な3つ」

私は「AEGIS」という140体のAIエージェント組織をClaude Codeで運用しています。プロダクト開発、コンテンツ制作、セキュリティ監査、経理まで、ほぼ全業務をAIと協働しています。

半年以上の運用で、AIが**安定して高い成果を出す領域**が明確になりました。

### 得意領域1: パターン認識と分類

- ログの分類・サマリー作成
- コードのlint違反・セキュリティ脆弱性の検出
- ドキュメントの構造化・要約

共通点は「既知のパターンに照合する」タスクです。正解が明確で、文脈依存が低い。

### 得意領域2: 定型的な生成

- テストコードの生成（既存パターンを踏襲）
- CRUDエンドポイントの追加
- 設定ファイルのテンプレート生成

「このファイルと同じ形式で作って」は、AIの最も得意なプロンプトです。

### 得意領域3: 網羅的チェック

- 全ファイルに対するセキュリティスキャン
- 依存ライブラリの脆弱性チェック
- コーディング規約の一貫性確認

人間が見落とす「漏れ」を防ぐ用途。ここはAIが人間を明確に上回ります。

**共通する特徴**: いずれも「判断」ではなく「照合・生成・網羅」です。

## 3. AIが確実に間違える4パターン

ここが本記事の核心です。140エージェント運用で繰り返し観測した、AIの失敗パターンを4つに整理しました。

### パターン1: 相関→因果の誤認

**Anthropicが公式に報告した問題そのもの。**

AIは「AのあとにBが起きた → AがBの原因」と推論します。しかし現実のシステムでは、AとBの両方がCという隠れた原因で同時に起きていることが多い。

**実例**: CI/CDパイプラインが失敗したとき、AIは「直前のコミットが原因」と断定しがち。実際には、並行して走っていた依存ライブラリの更新がテスト環境を壊していた。

### パターン2: コンテキストの欠落

AIは**今見えている情報だけ**で判断します。「先月このAPIの仕様を変更した」「このモジュールは来週リファクタ予定」「このエラーは無視していい」——チームの暗黙知にアクセスできません。

**実例**: AIがコードレビューで「この変数は未使用なので削除すべき」と指摘。実際にはフィーチャーフラグで無効化されているだけで、来週のリリースで有効化される予定だった。

### パターン3: タイミング判断の失敗

「いつ実行するか」の判断にAIは弱い。ログの時系列を読めても、「今このタイミングでデプロイすべきか」「このアラートは今すぐ対応が必要か」の判断は文脈依存。

**実例**: AIが「テスト全通過」を確認して本番デプロイを推奨。しかし、同時刻に別チームがDB migrationを走らせており、デプロイすれば競合する状況だった。

### パターン4: 自信過剰な回答

最も危険なパターン。AIは**「分からない」と言わない**。不確かな情報でも断定的に回答し、しかもそれが論理的に筋が通っているように見える。

**実例**: バグ調査でAIが「原因はXです。修正はこうです」と自信を持って回答。修正を適用したら別の場所が壊れた。AIは部分的な情報から「もっともらしい仮説」を作るのが上手すぎる。

---

### ここまでのまとめ

| AIが得意 | AIが苦手 |
|---------|---------|
| パターン照合 | 因果推論 |
| 定型生成 | コンテキスト依存判断 |
| 網羅チェック | タイミング判断 |
| 高速処理 | 不確実性の表明 |

**原則: AIには「整理」させて、「判断」は人間が行う。**

ここまでが前提知識です。次のセクションからは、この原則を**具体的にどう実装するか**——CLAUDE.md設計、hooksによる安全ゲート、エスカレーション階層の実装パターンを、コピペ可能なテンプレート付きで解説します。

## 4. 対処法：壊れない自動化の設計パターン

140エージェントが日常的に動く環境で、前述の4つの失敗パターンを防ぐために設計した仕組みを公開します。

### 設計原則: HALT（停止）+ エスカレーション

AIの自動化で最も重要な設計は「どこで止めるか」です。

私のシステムでは、全エージェントの行動を4段階に分類しています。

```
AUTO     → 低リスク・可逆。AIが自動実行。承認不要。
NOTIFY   → 中リスク。実行して日次レポートで報告。
CONFIRM  → 高リスク・不可逆。人間の承認を待つ。
HALT     → 最高リスク。即座に全停止し、人間に通知。
```

**ポイント**: ほとんどの自動化失敗は「止めるべきところで止めなかった」ことが原因です。AIに100%の判断を委ねるのではなく、「ここまでは任せる、ここからは人間」の線引きをシステムに組み込む。

### パターン1対策: 「因果誤認」を防ぐ — 複数仮説の強制

AIが原因を1つに断定するのを防ぐため、プロンプトレベルで「常に3つ以上の候補を出す」ルールを組み込みます。

```markdown
# CLAUDE.md に追加するルール例
## 障害分析ルール
根本原因を特定する際は、必ず以下の手順を踏むこと：
1. 相関している事象を3つ以上リストアップする
2. それぞれについて「因果関係がない可能性」を検証する
3. 最も確度の高い仮説を提示し、確度（高/中/低）を明記する
4. 確度が「中」以下の場合、人間のレビューを求める
```

これだけで、AIが「メモリが原因です」と断定する代わりに「メモリ、クエリパターン、依存ライブラリの3つが候補です。最も可能性が高いのはクエリパターン（確度: 中）」と出力するようになります。

### パターン2対策: 「コンテキスト欠落」を防ぐ — 構造化メモリ

Claude CodeのCLAUDE.mdファイルに「チームの暗黙知」を構造化して書き込みます。

```markdown
# CLAUDE.md に追加する例
## 現在の制約（毎週更新）
- featureX は 2026-04-01 まで凍結中（法務レビュー待ち）
- user_cache モジュールは来週リファクタ予定（削除しないこと）
- /api/v2/ のエンドポイントは後方互換テスト完了まで変更禁止
```

**更新頻度が鍵**です。古い情報はむしろ有害。私は毎日の自動フックで「前回更新から7日以上経った項目」を警告する仕組みを入れています。

### パターン3対策: 「タイミング判断」を防ぐ — 環境チェックゲート

デプロイや破壊的操作の前に、自動で環境状態をチェックするゲートを設けます。

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "command": "bash scripts/check-safe-to-deploy.sh \"$TOOL_INPUT\"",
        "description": "デプロイ系コマンドの前に環境チェック"
      }
    ]
  }
}
```

チェックスクリプトの中身:
- 他のデプロイが進行中でないか
- DB migrationが走っていないか
- 直近のテスト結果が全通過しているか
- 現在時刻がデプロイ禁止時間帯でないか

1つでも引っかかったらexit 2で実行をブロックします。

### パターン4対策: 「自信過剰」を防ぐ — 確信度スコアの強制

AIの出力に確信度を付けさせ、閾値以下なら自動エスカレーション。

```markdown
# CLAUDE.md ルール
## 確信度ルール
すべての技術的判断に確信度（0.0-1.0）を付与すること。
- 0.8以上: 自動実行OK
- 0.5-0.8: 実行前に根拠を提示し、承認を求める
- 0.5未満: 実行せず、代替案を3つ提示する
```

注意: AIが自己申告する確信度は絶対的な指標ではありません。しかし、「断定的に回答する」デフォルト動作を変えるだけで、致命的な誤判断を大幅に減らせます。

## 5. 実践テンプレート集

ここまでの対策を実際に適用するためのテンプレートを公開します。コピペしてあなたのプロジェクトに合わせて調整してください。

### テンプレート1: CLAUDE.md — 自動化ガバナンスセクション

```markdown
## AI自動化ルール

### 行動分類
- AUTO: ファイル読み取り、lint実行、テスト実行、フォーマット
- NOTIFY: ファイル作成・編集、依存追加、設定変更
- CONFIRM: デプロイ、DB操作、外部API呼び出し、ブランチ削除
- HALT: 本番環境操作、secret/credential操作、不可逆な削除

### 判断ルール
- 根本原因分析: 常に3候補以上。確度明記。中以下は人間レビュー
- 確信度: 全判断に0.0-1.0。0.5未満は実行停止
- コンテキスト: 「現在の制約」セクションを必ず参照してから判断

### 現在の制約（YYYY-MM-DD更新）
- [ここにチーム固有の制約を記載]
```

### テンプレート2: hooks — 安全ゲート設定

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "command": "bash .claude/hooks/safety-gate.sh \"$TOOL_INPUT\"",
        "description": "破壊的操作の検出とブロック"
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "command": "bash .claude/hooks/post-edit-check.sh \"$TOOL_INPUT\"",
        "description": "編集後のsecret漏洩チェック"
      }
    ],
    "Stop": [
      {
        "command": "bash .claude/hooks/session-summary.sh",
        "description": "セッション完了時の自動サマリー"
      }
    ]
  }
}
```

### テンプレート3: safety-gate.sh

```bash
#!/bin/bash
set -e
INPUT="$1"

# 破壊的コマンドの検出
DANGEROUS_PATTERNS=(
  "rm -rf"
  "drop table"
  "git push.*--force"
  "git reset --hard"
  "DELETE FROM"
  "docker system prune"
)

for pattern in "${DANGEROUS_PATTERNS[@]}"; do
  if echo "$INPUT" | grep -qi "$pattern"; then
    echo "HALT: 破壊的操作を検出しました: $pattern"
    echo "人間の承認が必要です。"
    exit 2
  fi
done

# デプロイ系コマンドの検出
if echo "$INPUT" | grep -qi "deploy\|publish\|release"; then
  # 環境チェック
  if [ -f ".deploy-lock" ]; then
    echo "HALT: 他のデプロイが進行中です"
    exit 2
  fi
fi

exit 0
```

## 6. 「任せていい/ダメ」判断チェックリスト

最後に、新しいタスクをAIに任せるか判断するためのチェックリストです。印刷して手元に置いておくと便利です。

### AIに任せてOK（AUTO/NOTIFY）

- [ ] 正解パターンが明確に定義できる
- [ ] 過去の類似例がコードベース内にある
- [ ] 失敗しても即座に元に戻せる（可逆）
- [ ] チーム固有の暗黙知が不要
- [ ] タイミング判断が不要（いつ実行しても同じ結果）
- [ ] 影響範囲がローカル（自分のファイル/ブランチのみ）

**全部チェックが付いたら → AIに任せてOK**

### 人間が判断すべき（CONFIRM/HALT）

- [ ] 「なぜそうなったか」の推論が必要
- [ ] チームメンバーの意図や計画を考慮する必要がある
- [ ] 実行タイミングが結果に影響する
- [ ] 失敗したら復旧に時間がかかる（不可逆/高コスト）
- [ ] 外部システムや他チームに影響する
- [ ] AIの回答に「確度: 中」以下が含まれる

**1つでもチェックが付いたら → 人間が最終判断**

### グレーゾーンの判断基準

迷ったら、この1問で判断してください。

> **「これが間違っていたとき、自分で5分以内に元に戻せるか？」**

YESなら任せる。NOなら自分で判断する。シンプルですが、これが最も実用的な基準です。

## まとめ

Anthropicが公式に認めたように、AIには構造的な限界があります。相関と因果の区別、暗黙知の活用、タイミング判断——これらは現在のLLMアーキテクチャでは根本的に難しい。

しかし、**限界を知った上で設計すれば、AIは圧倒的に強力な業務パートナー**になります。

この記事で紹介した対処法をまとめます。

1. **HALT設計**: 行動を4段階に分類し、リスクに応じて自動停止
2. **複数仮説強制**: 原因を1つに断定させない
3. **構造化メモリ**: チームの暗黙知をCLAUDE.mdに書き出す
4. **環境チェックゲート**: 操作前に自動で安全確認
5. **確信度スコア**: 不確かな判断を可視化

「AIにできること」を最大化し、「AIにできないこと」を人間が補う。この分業設計が、2026年の業務自動化の正解です。
